# Nexum Development Agent

This agent provides comprehensive guidance for building and maintaining the Nexum social media platform with professional standards.

## Project Overview

**Nexum** is a full-stack social media application with:
- **Frontend**: React + Vite (modern SPA)
- **Backend**: Node.js + Express (REST API)
- **Features**: Posts, Comments, Likes, Notifications, User Profiles, Authentication

## Architecture

### Frontend Stack
- React 18+ for UI components
- Vite for fast development and building
- Context API for state management (AuthContext)
- CSS for styling
- Components: Feed, Navbar, PostCard, Sidebar, RightPanel, MainLayout

### Backend Stack
- Express.js for REST API
- Authentication middleware (JWT-based)
- Database abstraction layer (config/db.js)
- MVC pattern with Controllers, Models, Routes, Middlewares

## Project Structure Rules

### Frontend (`/src`)
```
src/
├── App.jsx              # Main app component
├── main.jsx            # Entry point

```
backend/src/
├── app.js              # Express app setup
├── server.js           # Server entry point
├── init-db.js          # Database initialization
├── config/             # Configuration files (db.js)
├── controllers/        # Business logic controllers
├── middlewares/        # Express middlewares
├── models/             # Data models/schemas

## Development Standards

### Frontend Development
1. **Component Structure**
   - Functional components only
   - Use hooks for state management
   - Props drilling minimized with Context API
   - One component per file

2. **State Management**
   - AuthContext for authentication state
   - Local state with useState for component-level state
   - Keep state close to where it's used

3. **Styling**
   - Global styles in index.css
   - Component-scoped styles when possible
   - Maintain consistent spacing and color schemes

4. **API Integration**
   - Centralized API calls in /src/api
   - Error handling and loading states
   - Authentication tokens in headers

### Backend Development
1. **API Endpoints**
   - Follow RESTful conventions
   - Use appropriate HTTP methods (GET, POST, PUT, DELETE)
   - Consistent response format: `{ success: boolean, data: any, message: string }`

2. **Authentication**
   - JWT token-based authentication
   - Middleware validation on protected routes
   - Refresh token strategy for security

3. **Error Handling**
   - Centralized error middleware
   - Consistent error response format
   - Proper HTTP status codes

4. **Database**
   - Connection pooling via config/db.js
   - Model-based data access patterns
   - Transaction support for complex operations

## Common Development Tasks

### When Building New Features
1. Create API endpoint in backend (`routes/`, `controllers/`, `models/`)
2. Create React component in frontend (`src/components/` or `src/pages/`)
3. Connect frontend to backend via API client
4. Add error handling and loading states
5. Test authentication flows if applicable

### When Adding Authentication
1. Use existing auth.middleware.js for protected routes
2. Store JWT in frontend (localStorage or context)
3. Include token in request headers via api client
4. Handle token expiration and refresh

### When Adding New Data Models
1. Define schema in backend models
2. Create corresponding controller methods
3. Create routes for CRUD operations
4. Create API client function in frontend
5. Create React components to display/interact with data

## Code Quality Guidelines

### Naming Conventions
- Components: PascalCase (e.g., PostCard.jsx)
- Functions/variables: camelCase (e.g., handleSubmit)
- Constants: UPPER_SNAKE_CASE (if applicable)
- Route files: feature.routes.js, controller files: feature.controller.js

### Performance Considerations
- Lazy load components when possible
- Optimize API requests (avoid over-fetching)
- Use React.memo for expensive components
- Implement pagination for large lists

### Security Best Practices
- Validate and sanitize all user inputs
- Never expose sensitive data in frontend
- Use HTTPS in production
- Implement CORS properly
- Rate limiting on API endpoints

## Common Patterns

### Authentication Flow
1. User submits login credentials
2. Backend validates and returns JWT
3. Frontend stores token (context + localStorage)
4. Subsequent requests include token in Authorization header
5. Middleware verifies token on protected routes

### Post/Comment/Like Flow
1. User creates/interacts with content
2. Frontend sends request to backend
3. Backend validates, saves to database
4. Frontend updates UI optimistically or refetches data
5. Handle errors with user feedback

### User Profile
1. Fetch user data from backend
2. Display user posts, followers, following
3. Allow editing own profile (with authentication)
4. Show other users' profiles as read-only

## Common Issues & Solutions

### CORS Issues
- Configure CORS in Express: `app.use(cors())`
- Ensure frontend URL is in allowed origins
- Check credentials mode in fetch requests

### Authentication Failures
- Verify JWT secret matches between token generation and validation
- Check token expiration
- Ensure token is included in request headers

### Database Connection Failures
- Verify database is running and accessible
- Check connection string in config/db.js
- Ensure database user has proper permissions
